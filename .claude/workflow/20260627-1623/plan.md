## Ticket: SB-45 — Streak Motivation Nudges

Add an on-demand AI "streak motivation nudge": when a user has missed an active **daily** habit for **2+ consecutive days** (today and yesterday both lack a `completed = true` log), the dashboard surfaces a short, warm, personalised nudge generated by a new `nudge-agent.ts`. Implemented end-to-end mirroring the Goal Conflict / Daily Briefing pattern (agent → ai-config → index re-export → API route → dashboard card → dashboard page insertion).

### Decisions on PRD open questions / corrections

- **PRD correction — auth is Clerk, not NextAuth.** FR-4/AC-1 mention "NextAuth session"; this project uses **Clerk** via `requireUser()` from `@/lib/auth` (same as `goal-conflict/route.ts`). The route will authenticate with `requireUser()` and return 401 on failure.
- **OQ-1 / FR-7 / AC-5 rate limit — `localStorage` daily gate (client-side), MVP.** The card writes a `localStorage` key `sb_streak_nudge_dismissed:<YYYY-MM-DD>` (UTC date) when dismissed, and on mount **skips the fetch entirely** if today's key is present. This satisfies "once per day, persists across sessions/hard-refresh on the same browser" with **no DB table and no schema migration** (NFR-3). No server-side timestamp is stored (unlike goal-conflict's `aiGoalConflict` cache table — deliberately not replicated here to honour NFR-3).
- **OQ-3 — "2+ days" = at least 2 consecutive missed days.** A habit triggers a nudge when **both today and yesterday** have **no `completed = true` HabitLog**. (`>= 2` consecutive, not exactly 2.)
- **OQ-2 — multi-habit:** all broken habits are passed to the LLM in one context; the prompt asks it to weave them into a single 2–3 sentence message (briefing-agent style). `habits[]` lists every triggering habit name.
- **Out of scope (PRD):** only `frequency === "daily"` habits are considered (weekly habits excluded). No push/email, no DB persistence of nudge history, no snooze.
- **NFR-3 confirmed — NO schema migration.** Detection reads only existing `Habit` (`isActive`, `frequency`, `name`, `category`, `icon`, `streak`) and `HabitLog` (`date`, `completed`, `habitId`, `userId`) columns. No new model/column/index. `prisma db push` / `generate` NOT required.

## Implementation steps

1. **`packages/ai-core/src/ai-config.ts`** (~2 lines): add `| "streakNudge"` to the `AIFeature` union (after `"goalConflict"`), and add to `FEATURES`:
   `streakNudge: { anthropic: MODELS.fast, gemini: GEMINI_MODELS.fast, groq: GROQ_MODELS.fast, maxTokens: 400 }`.
   Rationale: a short warm nudge is the same shape/length as `briefing` (fast model, 400–600 tokens) — fast model is sufficient and cheapest.

2. **`packages/ai-core/src/agents/nudge-agent.ts`** (~110 lines, new). Mirror `goal-conflict-agent.ts`:
   - Inline interfaces:
     ```ts
     export interface NudgeHabit { name: string; category: string; icon: string; streak: number; }
     export interface NudgeContext { userName: string; habits: NudgeHabit[]; }
     export interface NudgeOutput { hasNudge: boolean; message: string; habits: string[]; }
     ```
   - `getMockNudge(ctx): NudgeOutput` — deterministic: if `ctx.habits.length === 0` → `{ hasNudge: false, message: "", habits: [] }`; else a warm message naming the habit(s) and `habits = ctx.habits.map(h => h.name)`, `hasNudge: true`.
   - `fmtHabits(ctx)` helper listing `- "name" | category | icon | lost streak: N`.
   - `generateStreakNudge(ctx): Promise<NudgeOutput>`:
     - `if (shouldMockAI()) return getMockNudge(ctx);`
     - `if (ctx.habits.length === 0) return { hasNudge: false, message: "", habits: [] };` (backstop — never call the LLM with nothing to nudge about).
     - Build prompt: warm, 2–3 sentences, reference the specific habit(s) by name, encouraging not preachy; return ONLY JSON `{"hasNudge": boolean, "message": string, "habits": string[]}` (no markdown/fences). Use `SYSTEM_PROMPT_BASE`.
     - `const raw = await chat(getChatConfig("streakNudge"), SYSTEM_PROMPT_BASE, prompt);`
     - Defensive parse (strip ```` ```json ````/```` ``` ````, `JSON.parse`); validate `message` is a non-empty string and `habits` an array of strings; on any failure or empty message **return `getMockNudge(ctx)`** (NFR-2 / AC-7: never throw to caller). Force `hasNudge` true only when `message` is non-empty.

3. **`packages/ai-core/src/index.ts`** (~2 lines): add
   `export { generateStreakNudge } from "./agents/nudge-agent";`
   `export type { NudgeContext, NudgeOutput, NudgeHabit } from "./agents/nudge-agent";`

4. **`apps/web/src/lib/habit-streak.ts`** (~45 lines, new) — detection helper:
   - Signature: `findBrokenStreakHabits(prisma, userId: string, today: Date, tz?: string | null): Promise<NudgeHabit[]>` (import `NudgeHabit` type from `@secondbrain/ai-core`).
   - Compute the two day windows with `userDayRange` from `@/lib/datetime`: `todayRange = userDayRange(today, tz)` and `yesterdayRange = userDayRange(<today − 1 day>, tz)`.
   - Fetch active **daily** habits: `prisma.habit.findMany({ where: { userId, isActive: true, frequency: "daily" } })`.
   - If none → return `[]`.
   - Fetch completed logs across both days in one query: `prisma.habitLog.findMany({ where: { userId, completed: true, date: { gte: yesterdayRange.gte, lt: todayRange.lt } }, select: { habitId: true } })` (single query, no N+1).
   - Build `Set<habitId>` of habits with **any** completed log in the last 2 days. A habit is "broken" when it is **not** in that set (no completed log today **and** yesterday → both missing ⇒ 2+ consecutive missed days, OQ-3).
   - Return broken habits mapped to `NudgeHabit { name, category, icon, streak }`.
   - Note: `HabitLog.date` is `@db.Date` (midnight UTC). Using `userDayRange` half-open ranges keeps this tz-correct and consistent with the dashboard's existing `userDayRange` usage. The 2-day window `[yesterday.gte, today.lt)` is what we scan.

5. **`apps/web/src/app/api/ai/streak-nudge/route.ts`** (~55 lines, new) — mirror goal-conflict route minus the DB cache:
   - `export const maxDuration = 60;` and a `TIMEOUT` symbol (same pattern).
   - `requireUser()` in a try/catch → 401 `{ error: "Unauthorized" }` on failure (AC: unauthenticated → 401).
   - `const today = new Date(); today.setHours(0,0,0,0);`
   - `const habits = await findBrokenStreakHabits(prisma, user.id, today, user.timezone ?? undefined);`
   - **Backstop:** if `habits.length === 0` → return `{ hasNudge: false, message: "", habits: [] }` without calling the AI (AC-2).
   - Build `NudgeContext { userName: user.name ?? "there", habits }`.
   - `Promise.race([ generateStreakNudge(ctx), timeout(50_000) ])`.
   - Return `NextResponse.json(output)` (the `NudgeOutput` directly — card reads `data.hasNudge` etc.). NOTE: unlike goal-conflict (`{ report }`), return the output flat so the card type-matches `NudgeOutput`.
   - **NFR-2 / AC-7 — never 5xx to the UI for AI failure:** wrap the generate/race in try/catch; on `TIMEOUT` or any error, return **`{ hasNudge: false, message: "", habits: [] }` with status 200** (silent fallback per PRD). Only `requireUser` failure returns a non-200 (401). (This diverges from goal-conflict's 500/504 deliberately — PRD NFR-2 says hide AI failures from the UI.)

6. **`apps/web/src/components/dashboard/streak-nudge-card.tsx`** (~90 lines, new) — client component:
   - `"use client"`. Props: none (self-contained; it queries on mount). Optionally accept `activeDailyHabitsCount?: number` to early-return null when 0 — **decision: no props**, keep it self-gating via the localStorage + fetch result to avoid widening the dashboard data query. (Simpler; matches "render nothing when hasNudge=false".)
   - State: `nudge: NudgeOutput | null`, `dismissed: boolean`, `loading`.
   - localStorage key helper: `const key = (d: Date) => \`sb_streak_nudge_dismissed:${d.toISOString().slice(0,10)}\`;`
   - On mount (`useEffect`, `[]`): if `localStorage.getItem(key(new Date()))` is set → do nothing (gate, AC-5). Else `fetch("/api/ai/streak-nudge", { method: "POST" })`, set `nudge`. On error, swallow (no toast spam — nudge is non-critical; matches silent-failure philosophy).
   - Render: return `null` if `dismissed`, or `!nudge`, or `nudge.hasNudge === false`. Otherwise a `glass rounded-xl p-6 animate-fade-in` card with a lucide icon (e.g. `Flame` in amber, matching the dashboard streak motif), title "Keep your streak alive", `nudge.message`, and the affected habits as chips/list (`nudge.habits` — names; icons available if we widen, but `NudgeOutput.habits` is names only per FR-2, so render names; optional: look up icons from context is not available client-side — render names only).
   - "Got it" button: `setDismissed(true)` and `localStorage.setItem(key(new Date()), "1")` (AC-4 session dismiss + AC-5 daily persistence). Use `Button` from `@/components/ui/button`.
   - Guard all `localStorage` access for SSR safety (`typeof window !== "undefined"`); component is client-only so effect runs client-side — fine.

## DB changes required?

**No.** Streak detection derives entirely from existing `Habit` (`isActive`, `frequency`, `name`, `category`, `icon`, `streak`) and `HabitLog` (`date`, `completed`, `habitId`, `userId`) columns. The once-per-day rate limit uses **client `localStorage`**, not a DB table — so unlike `goal-conflict` (which added an `aiGoalConflict` cache table) **no new model/column/index** is introduced. No `prisma db push` / `prisma generate` needed (NFR-3 confirmed).

## New packages required?

**None.** Uses existing deps: `@secondbrain/ai-core`, `@secondbrain/db` (Prisma), `@/lib/auth` (Clerk `requireUser`), `@/lib/datetime` (`userDayRange`), `@/components/ui/button`, `lucide-react`.

## Test strategy

- **Unit — `nudge-agent.ts`** (ai-core, Vitest, `MOCK_AI=true` and mocked `chat`):
  - mock mode returns well-formed `NudgeOutput` with `hasNudge:true` and habit names when habits present (AC-6); `hasNudge:false` / empty when `habits=[]`.
  - LLM happy path: mocked `chat` returns valid JSON → parsed `NudgeOutput`; `habits[]` populated; message non-empty.
  - LLM failure / unparseable JSON → falls back to `getMockNudge` (no throw) (AC-7).
- **Unit — `habit-streak.ts`** (apps/web, mocked Prisma `habit.findMany`/`habitLog.findMany`):
  - habit with a completed log today OR yesterday → NOT broken.
  - habit with no completed log on either day → broken (returned) (OQ-3, AC-1).
  - inactive / non-daily habits excluded; empty active set → `[]`.
  - query scoped to `userId`, `completed:true`, correct 2-day date range; single `habitLog.findMany` call (no N+1).
- **Integration — `POST /api/ai/streak-nudge`** (apps/web, mirror `goal-conflict-route.test.ts`; mock `@secondbrain/ai-core.generateStreakNudge`, `findBrokenStreakHabits` or underlying prisma, `requireUser`):
  - 401 when `requireUser` throws.
  - no broken habits → `{ hasNudge:false }` 200 **without** calling the agent (AC-2 backstop).
  - happy path → returns agent `NudgeOutput` flat; `userName` falls back to "there" when `user.name` null.
  - agent throw / timeout → `{ hasNudge:false }` **200** (NFR-2/AC-7), not 5xx.
- **Smoke — `streak-nudge-card.tsx`** (apps/web, Testing Library): renders nothing when fetch returns `hasNudge:false`; renders message + habits when `hasNudge:true`; "Got it" hides the card and writes the localStorage key; mounting with today's localStorage key set → no fetch, renders nothing.
- `apps/web/src/__tests__/setup.ts` already mocks `habit.findMany` and `habitLog.findMany` — sufficient; no setup edit expected.

## Risk / unknowns

- **`HabitLog.date` storage vs. tz:** `date` is `@db.Date` (midnight, no time). The dashboard already filters "today" with `prisma.habitLog.findMany({ where: { date: today } })` using a local-midnight `Date`, while tasks use `userDayRange`. For the 2-day window I use `userDayRange` half-open ranges for tz-correctness; the Dev Agent should confirm against how logs are *written* (the `/api/habits` log route) so detection aligns with how "completed today" is recorded. Low risk — half-open UTC range covering both calendar days is robust either way.
- **Missing-log semantics:** absence of a `completed=true` log = "missed". A habit created today legitimately has no yesterday log and would read as a 2-day break. **Mitigation:** acceptable for MVP (a habit with `streak: 0` and recent `createdAt` is unlikely to motivate a nudge); if noisy, a follow-up can filter `habit.streak > 0` or `createdAt` older than 2 days. Not added now to keep scope minimal — noted for reviewer.
- **localStorage rate limit is per-browser/per-device only** (OQ-1 accepted limitation): switching device/browser re-shows the nudge the same day. Acceptable for MVP.
- **Flat response shape divergence from goal-conflict** (`NudgeOutput` directly vs `{ report }`) and **silent-200-on-failure divergence** are intentional per PRD FR-2/NFR-2 — reviewer should not flag as inconsistency.
